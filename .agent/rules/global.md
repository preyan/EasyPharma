# EasyPharma Global Rules

- **LTS Strictness**: Strictly use Angular 21.x and NestJS 11.x (Express). No experimental versions.
- **Architecture**: Nx Monorepo with Domain-Driven Design (DDD).
- **ORM**: Prisma 7 (Schema-first).
- **Testing**: Use Vitest for Unit/Integration and Playwright for E2E. Minimum 80% coverage.
- **Documentation**: Swagger (API) and Compodoc (Frontend) must be autogenerated.
- **Workflow**: Commit small, atomic changes frequently to the main branch using Conventional Commits.
- **Autonomy**: Agent is authorized to run 'pnpm install', 'nx g', and 'prisma' commands.

# Terminal Execution Rules

- **Environment**: Windows 10/11 using Git Bash (MINGW64).
- **Shell Constraint**: NEVER use 'cmd.exe' or 'powershell.exe'.
- **Command Syntax**: All terminal commands must be formatted for Bash (e.g., use 'ls' instead of 'dir', forward slashes for paths).
- **Execution**: If the agent needs to run a complex script, it must create a .sh file and execute it via 'bash script.sh'.
- **Persistence**: Do not use 'cmd /c'. Use native bash execution to maintain the environment state.

# Directory & Execution Rules

- **Scripts**: ALL bash scripts must be stored in `/scripts`. Never execute scripts directly from the root or app folders.
- **Shell**: Strictly use Git Bash (MINGW64). Forbidden: cmd.exe, powershell.exe.

# Agent Self-Monitoring & Autonomy

- **Stuck Command Detection**: If a command exceeds 60 seconds without output, kill the process, run `nx reset`, and diagnose.
- **Loop Prevention**: Never run the same command with identical parameters more than twice. If it fails twice, STOP and propose a workaround.
- **Context Awareness**: After every command, output current usage:
  > [Context Usage: X tokens / Y max]
- **Idempotency**: Before running `pnpm install` or `nx g`, check if the target files already exist to avoid redundant operations.
- **Full Log Capture**: If a command fails and the error output is longer than 20 lines, the agent MUST redirect the output to a temporary log file (e.g., `npx nx build > logs/error.log 2>&1`).
- **Stack Trace Inspection**: The agent is forbidden from guessing the error if the terminal output is truncated. It must read the generated log file from the beginning to find the root cause.
- **Log Cleanup**: After resolving the issue, the agent must delete the temporary log file to keep the `/logs` directory clean.

# Anti-Loop & Error Recovery

- **State Verification**: Before retrying any failed command, the agent MUST run `ls -a` or `git status` to check if any files were partially created.
- **Back-off Strategy**: If a command fails twice, the agent MUST:
  1. Delete the `node_modules` or `.nx/cache` (if relevant).
  2. Run `pnpm nx reset`.
  3. Explain to the user exactly what error code was received.
- **Manual Intervention**: If a fix requires changing a system-level environment variable not in the `/scripts` folder, the agent MUST stop and ask the user to do it.
- **Script Usage**: Before starting any Phase, run `/scripts/health-check.sh`.
- **Committing**: ALWAYS use `/scripts/atomic-commit.sh "[message]"` to ensure versioning and changelogs are handled.
- **Troubleshooting**: If a command fails twice, run `/scripts/clean-slate.sh` before the third attempt.

# Test Execution & Loop Prevention

- **The Three-Strike Rule**: If a test suite fails 3 times in a row, the agent MUST STOP. It is forbidden to run the test a 4th time without a manual code intervention or a strategy pivot.
- **Isolation Protocol**: If a test fails, the agent must run that specific test in isolation (e.g., `nx test app --testFile=src/lib/logic.spec.ts`) instead of the entire suite to reduce noise.
- **Cache Invalidation**: On the second failure, the agent must run the test with `--skip-nx-cache` to ensure it's not looking at stale "passed" results.
- **Deep Inspection**: Before retrying a failing test, the agent must `cat` the spec file and the source file simultaneously to verify the logic alignment.

# Issue Tracking & Traceability (GitHub CLI)

- **Automatic Issue Creation**: If an error persists after 2 repair attempts, or if a significant architectural blocker is found, the agent MUST create a GitHub Issue using `gh issue create`.
- **Template Compliance**: The agent MUST use `.github/ISSUE_TEMPLATE/bug_report.md`. All sections (Steps to Reproduce, Expected vs Actual, etc.) must be populated using current session data.
- **Log Management**: Do not paste raw logs over 50 lines. Use `grep -C 5` to extract the relevant failure point or summarize the stack trace to avoid cluttering the issue.
- **Labeling**: Every issue must be created with the `--label "agent-reported"` flag, plus one domain label (e.g., `area:nx`, `area:infra`, `area:tests`).
- **Commit Referencing**: Every commit intended to fix an issue MUST include `Fixes #<IssueNumber>` or `Refs #<IssueNumber>` in the FIRST line of the commit message.
- **State Sync**: Update the local `plan.md` task by appending the GitHub Issue URL next to the task item for immediate visibility.
- **Self-Healing AI Oversight**: If 'Nx Self-Healing' suggests a fix for a test, the agent must verify the fix doesn't use `any` types or bypass security guards to make the test pass.
